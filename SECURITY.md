# Security model

This is the long-form companion to the security notes in [README.md](README.md).
It states what VanishDrop defends against, what it intentionally doesn't,
and the cryptographic reasoning behind both.

The goal is to let a security-minded reader audit the project end-to-end
in a single sitting.

---

## Cryptographic primitive

VanishDrop uses the browser's **Web Crypto API** with AES-GCM:

- 256-bit symmetric key (generated by `crypto.getRandomValues`).
- 96-bit (12-byte) IV (fresh per encryption, generated by `getRandomValues`).
- AES-GCM authenticated encryption — the 128-bit authentication tag is
  appended to the ciphertext in the standard Web Crypto layout.

This is a NIST-standardised, hardware-accelerated AEAD on every modern
platform. The Web Crypto API is exposed under `window.crypto.subtle`
in all evergreen browsers and on Node 19+.

### Key derivation

There is no key derivation. The 32-byte key is generated by
`crypto.getRandomValues(new Uint8Array(32))` in the browser, exported
as raw bytes, and embedded in the URL fragment. No passphrase, no
PBKDF2.

### Nonce reuse

Each upload generates a fresh 12-byte IV. Reuse of a (key, IV) pair
under AES-GCM is catastrophic, but each key is one-shot — generated
fresh per upload, used exactly once, then garbage-collected. The
collision probability for two random keys colliding is negligible.

## Threat model

### 1. Honest-but-curious server operator

**Defended.** The server stores only `{ ciphertext blob, iv, filename,
expires_at, downloads_remaining }`. The key is in the URL fragment —
the browser never sends fragments in HTTP. The server cannot decrypt,
even with full disk and log access.

What the operator can see:
- That a file existed.
- Its size, original filename, TTL, max-downloads.
- When it was uploaded; when (and from what IP) it was downloaded.

What is hidden:
- The plaintext.
- The decryption key.

### 2. Passive network observer

**Defended, conditional on TLS.** VanishDrop must be deployed behind
HTTPS. With TLS, the observer sees only encrypted bytes plus IP-level
metadata. Without TLS, AES-GCM still holds for confidentiality, but
filename + size + IPs are exposed.

### 3. Active MITM

**Defended with TLS pinning of the host.** A MITM cannot decrypt
without the URL-fragment key, and AES-GCM's authentication tag rejects
any tampering. A MITM that intercepts the *response* could substitute
their own ciphertext, but the recipient's decrypt would fail with a
generic "could not decrypt" rather than a falsified plaintext.

### 4. URL leakage (the realistic risk)

**Partially defended.** A capability URL is only as safe as its
distribution channel:

- **Browser history**: the full URL, fragment included, is stored
  locally. Once consumed, the server copy is gone — the URL in history
  becomes worthless.
- **Referer header**: clicking outbound links from the viewer page
  would leak the URL via Referer. The SPA shell uses
  `<meta name="referrer" content="no-referrer">` to prevent that.
- **Server access logs**: HTTP servers cannot log fragments by
  construction.
- **Chat/email middlemen**: anyone who can read the channel you send
  the URL through can download the file. Use a private channel.

### 5. Compromised browser / XSS

**Best-effort defence.** Mitigations:

- Strict Content-Security-Policy applied by the FastAPI middleware:
  `default-src 'self'`, `script-src 'self'`, `connect-src 'self'`,
  `frame-ancestors 'none'`.
- No third-party JavaScript origins (Vue + the app bundle are served
  from the same origin).
- Vue 3 escapes interpolations by default; we never use `v-html` with
  user content.

### 6. Compromised server delivering a backdoored bundle

**Not defended in code.** Like any web-delivered E2E tool, the server
ships the JavaScript that does the encrypting. A compromised server
could deliver a bundle that exfiltrates the fragment key. Standard
mitigations (code signing, reproducible builds, browser extensions)
are out of scope for this demo.

### 7. Large-file DoS

**Bounded.** Server caps blob size at 32 MiB and per-record TTL at 7
days. The SQLite DB and the on-disk blob directory grow predictably
under load; a sweep job (every 60 s) removes expired records.

## Server-side validation

The server does not trust client input. Every value is checked before
state mutation:

| Field           | Rule                                                            |
| --------------- | --------------------------------------------------------------- |
| `iv`            | Base64-decodes to exactly 12 bytes (AES-GCM IV size).           |
| `filename`      | ≤ 255 chars; no `/`, `\`, or NUL.                                |
| `ttl_seconds`   | Clamped to [60, 604800] (1 minute to 7 days).                    |
| `max_downloads` | Clamped to [1, 100].                                             |
| Blob size       | Streamed-write capped at 32 MiB; over-size aborts mid-stream.    |
| `id` (path)     | 20–32 chars, urlsafe base64 alphabet.                            |

The atomic download-consume + record-delete logic lives entirely
inside the `GET /api/files/:id` handler — there is no client-side
enforcement of read count.

## Intentional non-goals

- **No DELETE / edit / list endpoints.** Once posted, a file cannot
  be retracted, modified, or enumerated by the creator. It dies by
  being downloaded or by expiring.
- **No download receipts.** The creator cannot learn whether their
  file was downloaded.
- **No passphrase protection.** Could be added (PBKDF2 / Argon2 over
  a user-supplied passphrase, wrap the random key); not in v1.
- **No multi-recipient or share-with-N-named-people.** The model is
  one URL = one capability.
- **No federation.** Each VanishDrop instance is its own namespace.

## Reporting a vulnerability

Please **do not** open a public issue. Email the maintainer first so a
fix can be coordinated before disclosure. See
`/.well-known/security.txt` for the contact and policy URL.
